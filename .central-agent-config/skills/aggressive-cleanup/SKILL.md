# Aggressive Cleanup

## Purpose

Make NahidArbX smaller, clearer, and easier to maintain by aggressively removing noise and duplication while preserving runtime behavior.

This skill is intentionally high-agency: do not stop at suggestions when the cleanup can be executed safely. Inventory, edit, verify, and remove leftovers.

## Operating Rules

- Work in `/Users/nahidhasan/Projects/nahidArbX`.
- Read `AGENTS.md` before changing files.
- Treat the current branch as the working branch. Do not create branches.
- Respect existing user changes. Do not revert unrelated dirty worktree changes.
- Keep cleanup behavior-preserving unless the user explicitly asks for a product behavior change.
- Prefer repository patterns over introducing new frameworks or stylistic rewrites.
- Delete dead code instead of leaving compatibility wrappers unless an external contract still needs them.
- After code changes, run focused tests where relevant, then `npm run build` and `npm run lint`.
- Clean up temporary scripts, scratch files, generated artifacts, and unused imports before finishing.

## Cleanup Targets

### Remove Code Comments

- Remove all explanatory, TODO, stale, section-header, and commented-out-code comments from source files.
- Do not add new code comments.
- Replace `eslint-disable`, `ts-ignore`, `ts-expect-error`, coverage-ignore, and similar directive comments with real fixes whenever possible.
- Preserve only syntax-critical non-comment directives such as `"use client"` and `"use server"`.
- Preserve shebang lines only for executable scripts that still need direct shell execution.
- If a comment appears legally required or generated by a tool, prefer moving that requirement to build/tool configuration. If it truly cannot be removed, report the exact file and reason.

### Refactor Duplication

- Search for repeated logic before editing. Use `rg`, `rg --files`, tests, and type references.
- Extract reusable functions only when the shared behavior is real and stable.
- Prefer small pure helpers in the nearest appropriate domain module.
- Move cross-cutting helpers to existing shared locations such as `lib/shared`, `lib/util`, or the owning domain package.
- Keep domain boundaries clear: settlement logic stays under `lib/settle`, matcher logic under matcher modules, ML accounting under ML modules, provider specifics under provider folders.
- Do not create generic abstractions that hide important business rules or make call sites harder to read.

### Architecture And System Design

- Keep background work in `engine.ts` and background modules, not Next.js boot or UI code.
- Keep Next.js API/UI thin: validation, authorization/session concerns, repository calls, and response shaping only.
- Keep database access behind repositories or established DB helpers.
- Use Zod for external data boundaries.
- Use `singleton()` for HMR-safe process state.
- Keep constants in `lib/shared/constants.ts` when a value is shared or business-significant.
- Use dependency direction that matches the domain: UI imports API clients/types, APIs import services/repositories, services import domain logic, domain logic avoids UI/API imports.
- Prefer explicit pure functions, repository boundaries, adapters, strategy maps, and pipeline stages over hidden global state.
- Collapse unnecessary layers when they only pass arguments through.

### Remove Markdown Documentation

- Delete stale or redundant repository documentation Markdown files aggressively.
- Preserve operational instruction files required by agents and skills: root `AGENTS.md`, compatibility `CLAUDE.md` symlink if present, and every skill `SKILL.md`.
- Preserve Markdown that is package-required or externally consumed only when deleting it would break tooling, publishing, or agent discovery. Report any preserved exception.
- When deleting docs, migrate any still-true operational rule into `AGENTS.md` or the relevant skill only if it is essential and not already covered.
- Do not create replacement docs unless the user explicitly asks.

### Remove Stale Scripts

- Inventory `scripts/`, one-off SQL, migration helpers, demo scripts, and ad hoc backfill/resettle files.
- Delete scripts that are obsolete, one-time, unreferenced, superseded by package scripts, or dangerous without current safeguards.
- Keep scripts that are referenced by `package.json`, CI, migrations, production workflows, or active developer commands.
- If a script is useful but misplaced, move it into the nearest domain folder or formalize it behind a package script.
- Delete related stale tests, fixtures, and snapshots only when the tested behavior no longer exists.

### Organize Folder Structure

- Move files toward ownership-based folders, not miscellaneous buckets.
- Keep route components with their route or established component domain.
- Keep reusable UI under `components/ui` only when generic enough for multiple surfaces.
- Keep domain UI under the owning feature folder.
- Update imports with the existing alias style.
- Remove empty folders after moves.
- Avoid churn-only moves when names and ownership are already clear.

## Workflow

1. Inspect the dirty worktree.
   - Run `git status --short`.
   - Identify unrelated user changes and avoid overwriting them.

2. Build an inventory.
   - Comments: use `rg` for `//`, `/*`, `*/`, `#`, `TODO`, `FIXME`, `eslint-disable`, `ts-ignore`, and `ts-expect-error`.
   - Duplication: search repeated function bodies, repeated constants, similar query shaping, repeated formatters, repeated request/response handling, and copied business rules.
   - Docs: list `*.md` files and classify as instruction, active operational, stale docs, or package-required.
   - Scripts: list `scripts/` and root executable files; cross-check `package.json`, imports, tests, and references.
   - Structure: identify files whose ownership is unclear or whose imports create backwards dependencies.

3. Plan a small set of coherent edits.
   - Prefer one cleanup theme at a time: comments, duplicated domain logic, stale docs/scripts, then folder moves.
   - For large cleanups, keep edits reviewable by domain rather than touching every file blindly.

4. Edit aggressively.
   - Delete stale files.
   - Remove all comments in touched code.
   - Extract shared helpers where duplication is meaningful.
   - Simplify call sites after extracting helpers.
   - Update imports, exports, tests, and package scripts.

5. Verify behavior.
   - Run focused tests for touched domains.
   - Run `npm run build`.
   - Run `npm run lint`.
   - If verification fails, fix the cleanup rather than leaving new failures.

6. Final report.
   - Summarize removed comments/docs/scripts and major refactors.
   - List preserved Markdown or comments with reasons.
   - Report verification commands and results.
   - Mention remaining cleanup opportunities only when they are concrete and not safe to do in the current pass.

## Decision Heuristics

- Delete over deprecate when code has no active caller.
- Extract over duplicate when at least two call sites share business logic, validation, normalization, formatting, query construction, or side-effect orchestration.
- Inline over abstract when a helper would only hide one obvious expression.
- Move over rename-only churn when folder ownership is wrong and import direction improves.
- Keep over delete when a file is referenced by build, package scripts, production jobs, migrations, or required agent instructions.
- Fix over suppress when a directive comment exists.
- Test over trust when cleanup touches settlement, matching, betting placement, notifications, ML training/accounting, database writes, or provider adapters.
