# AGENTS.md

This file provides guidance to agents when working with code in this repository.

## Keep in sync with CLAUDE.md

This file and [`CLAUDE.md`](CLAUDE.md) are intentional mirrors. AGENTS.md is a terse index; CLAUDE.md is the full-form reference. Entries here point back with "See CLAUDE.md 'X' section" lines. **When you add, change, or remove a rule in either file, update the other in the same commit.** A new CLAUDE.md rule needs a one-liner here that cross-references it; a new AGENTS.md entry needs a full-form section in CLAUDE.md. Drift between the two is a documentation bug.

## Build & Test Commands

- `npm run build` — production build (always run after changes, per CLAUDE.md rules)
- `npm run lint` — ESLint (next/core-web-vitals + typescript configs)
- `npm run test:unit` — Node built-in test runner via `node --import tsx --test lib/**/*.test.ts` (NOT vitest)
- `npx vitest run` — Vitest unit tests from `tests/unit/` (separate from test:unit above)
- `npm run test:settle` — Node runner focused on `lib/settle/*.test.ts`
- Single unit test: `node --import tsx --test lib/settle/settle-bet.test.ts`
- DB: `npm run db:generate` then `npm run db:migrate` (drizzle-kit, snake_case casing)

UI verification is manual — the user exercises changed features in-browser. Do not run Playwright E2E suites.

## Critical Non-Obvious Rules

- **Use `singleton()` from `lib/util/singleton.ts`** for any state that must survive Next.js HMR / Turbopack's separate module graphs. Module-level `let` gives independent copies per loader.
- **`instrumentation.ts`** is the server-boot hook that starts the sync + auto-settlement schedulers — the system is headless and runs whether or not the UI is open.
- **The auto-settle kill-switch was removed in 2026.** No `lib/settle/kill-switch.ts`, no `/ai` Telegram command, no kill-switch UI button. There's nothing to engage because automatic Gemini AI usage was removed entirely — settlement is deterministic Tier 0/1/2 only. See CLAUDE.md "AI usage policy — manual only, no automatic Gemini" section.
- **The ONLY paid Gemini surface is operator-triggered:** the `/bets` "AI settle" dropdown calls `aiLabelBets(ids, { forceAi: true, aiModel })` which is the only path to Tier 3 `url_context`. The automatic settlement scheduler MUST never set `forceAi: true`. Pre-flight cost-guard refuses batches above `AI_MAX_PER_REQUEST_USD` (default $2).
- **Bet IDs are deterministic**: `${eventId}|${familyId}|${atomId}` (not UUIDs) — primary key for `bets` table.
- **`bets` is the only app/backtest settlement table** — `value_bets` and `placed_bets` are dropped legacy tables. New code must use the merged `bets` table everywhere, and cleanup work should remove stale references to the old split-table model.
- **Settlement pipeline must be shared everywhere** — auto-settle, manual re-settle, and manual outcome application should converge on `settleBatch` / `applySettlementOutcomes` rather than drifting into separate code paths.
- **Telegram settlement notifications are only for actually placed bets** — if a row was never placed (`placedAt` is null), settle it silently with no notification.
- **Auto-place stakes snap to multiples of 100 BDT** (`AUTO_PLACE_STAKE_BUCKET`), min 200 BDT. Never submit fractional stakes.
- **`better-sqlite3` is auth-only** — do NOT use it for app data. App DB is Postgres via Drizzle.
- **Do not assume the local `.env` `DATABASE_URL` or `127.0.0.1` Postgres is the source of truth** — when runtime DB inspection is needed, prefer the real cloud database path via the app's configured SDK/connector. Document any local-only assumptions before using them.
- **Middleware uses `jose`** (Edge Runtime compatible), not `jsonwebtoken`.
- **Drizzle casing is `snake_case`** — DB columns are snake_case, TS code uses camelCase via Drizzle's casing transform.
- **`lib/shared/constants.ts`** is the single source of truth for magic numbers (not `lib/config.ts`).
- **`lib/betting/settlement-cascade.ts`** is deprecated (empty re-export) — settlement is inline in bets table now.
- **Two separate test systems**: Vitest (`tests/unit/`) and Node built-in runner (`lib/**/*.test.ts`). `npm run test:unit` uses the latter.
- **Single `.env` file** — this project uses one `.env` at repo root. Do not create `.env.local` or `.env.example`.
- **Runtime is localhost.** The production domain (`nahidarbx.store`) is no longer active. The app runs locally via `npm run dev` (http://localhost:3000). Verify UI changes at `http://localhost:3000`. The Python optimizer sidecar (`nahidarbx-optimizer`) runs on Cloud Run (`nahidarbx-6e73`, region `asia-south1`) — redeploy via `bash services/optimizer/redeploy.sh`. Cloud SQL Postgres is still the app database; local dev requires `cloud-sql-proxy --port 5432 nahidarbx-6e73:asia-south1:nahidarbx-db`.
- **Cloud SQL Postgres** in production (`nahidarbx-6e73:asia-south1:nahidarbx-db`). Local dev requires `cloud-sql-proxy --port 5432 nahidarbx-6e73:asia-south1:nahidarbx-db`.
- **Gemini cost safety** — paid `url_context` calls are gated by per-batch cost ceiling (`AI_MAX_PER_REQUEST_USD`, default $2) AND fire only when `opts.forceAi === true`. The automatic scheduler never passes that flag. See CLAUDE.md "AI usage policy — manual only, no automatic Gemini" section for the full protocol.
- **Tooltips are foundational — every meaningful control has one.** Use the `<Tooltip>` / `<TooltipTrigger>` / `<TooltipContent>` primitives from `components/ui/tooltip.tsx` — never plain `title=""` for actionable controls (it fails on touch and can't render rich content). Wrap the panel root in `<TooltipProvider delayDuration={200}>` once. Every workflow step (tab, filter chip, action button, bulk dropdown, status badge) gets a tooltip explaining what it does and what happens next. State-aware tooltips must reflect the current state — don't ship a tooltip that lies about what the click will do. Any control that triggers a billed AI call (Gemini Lite/Flash/Pro) must say "AI calls cost money" in the tooltip and recommend cheaper paths first. The Matcher Lab's pre-2026-04-26 reliance on `title=""` is the cautionary tale this rule exists for. See CLAUDE.md "Tooltips are foundational" section.
- **Entity Resolution** — alias system is Postgres-backed (4 tables: `entities`, `entity_names`, `name_observations`, `entity_review_queue`, `entity_resolver_runs`), tournament-scoped (`(provider, surface_normalized, competition_id)` UNIQUE), with a 4-tier ML promoter (deterministic gates → Bayesian evidence → LightGBM + conformal calibration → operator review) and a weekly Splink+Leiden graph-cleanup Cloud Run Job. Every alias-learning writer (matcher, settle, match-review, learner) calls one ingress `recordObservation` in `lib/matching/entities/observations.ts`. Tier 0 gates include gender mismatch, **team-variant mismatch (U17/U19/U20/U21/U23/Sub-20/Olympic/Reserves/II/B/Castilla/Academy/Futsal/Beach/eSports/Selects/Youth)**, group conflict, and competing-candidate. The legacy `data/aliases/*.json` + `AliasManager.tsx` were deleted. UI replacement is `EntityInspector` at `/diagnostics → Entities` — seven tabs all built on `<DataTable>`: Overview (KPIs + sparkline + classifier histogram + active-Job card), Entities (drawer for surface forms), Surface forms (candidate-first), Observations (live audit log), Review queue (Splink/Leiden findings), **Job runs (live progress card while a Cloud Run Job is in flight, polling every 2 s)**, Playground (resolver/classifier probe + observation submitter). See CLAUDE.md "Entity Resolution — alias system" section.
- **Solo-developer workflow — no branches.** Personal project; everything lands on the working branch (treat as master). Don't propose feature branches, don't fragment "my changes" vs "user's WIP" when committing. When asked to commit, commit everything that's relevant in as few commits as the work warrants. See CLAUDE.md "Solo-developer workflow" section.
- **Cloud Run: Jobs for batch work, Service only for HTTP receivers.** If a task takes more than a few seconds and isn't tied to an HTTP request, deploy as a Cloud Run **Job** (one execution per task, no autoscaler-reap, 7-day max). `--no-cpu-throttling` does NOT keep a Service instance alive without in-flight requests — the only Service-side fix for long background work is `--min-instances=1`, which burns ~$80–100/mo. The Optimisation sidecar was migrated from Service → Job on 2026-04-25 after a sweep died mid-flight. See CLAUDE.md "Cloud Run lesson" section.
- **Toolbar / filter components must be reused across spreadsheet surfaces.** When adding a new list/table page, reuse the patterns in `BetsHistoryToolbar` and `SpreadsheetToolbar` — same control heights (`h-7`), same `px-3 py-1.5` wrapper, same `bg-muted/40` backdrop, same `text-[11px]` buttons. If you need a new variant, extract the common parts into a shared component first. Inconsistent sizing between pages was a known pain point (April 2026).
- **Every table uses `<DataTable>`** (`components/ui/data-table.tsx`) — virtualization, sorting, column resize, drag-to-reorder, persistence, infinite scroll, grouping, selection are all opt-in via flags. Do NOT write a new plain `<table>` for tabular data; if a feature is missing from `<DataTable>`, extend it in place rather than forking. A 500-row plain table on `/lab/optimisation/[id]` was the root cause of laggy polling on 2026-04-25. Always pass `getRowId` when rows come from a polled query so the virtualizer doesn't rebuild on every poll. Only existing exception: `components/spreadsheet/ValueBetSpreadsheet.tsx` (positional row layout, doesn't fit a flat column model). New tables are not allowed to become further exceptions. See CLAUDE.md "Reusable table component" section.
- **Styling is Tailwind only — no custom CSS.** Every layout / color / animation / hover state goes on the JSX as Tailwind utilities (or arbitrary-value variants like `bg-[oklch(...)]` / `grid-cols-[repeat(24,minmax(0,1fr))]`). If a long combo repeats, lift it into a React component, NOT a CSS class. `app/globals.css` is reserved for Tailwind imports, the `@theme inline` token map, `:root` / `.dark` theme variables, the `@layer base` border reset, Sonner toast overrides, and global scrollbar styling. Component-scoped class blocks (`.db-*`, `.acc-*`, `.appshell-*`, `.fintech-*`, `.metric-pod`, `.glass-panel`, etc.), app-specific keyframes, and utility-helper classes (`.data-text`) are NOT allowed — Turbopack silently dropped a 400-line custom-CSS block on 2026-04-25 and broke the dashboard layout. See CLAUDE.md "Styling — Tailwind only" section for the migration recipe and what's whitelisted in `globals.css`.
- **Typography tiers — prose vs. chrome.** Every bit of text is one of two tiers. **Prose** (sentences a user has to read — tooltip bodies, description paragraphs, help captions, empty states, error messages, section intros) → **`text-sm` / 14px minimum** (`text-[13px]` acceptable for secondary captions). **Chrome** (labels users scan — toolbar buttons, pill filters, table/spreadsheet cells, badge counters, stat numerals) → `text-[11px]` / `text-xs` per the April-2026 toolbar-consistency rule. Heuristic: inside `<Button>`/`<Badge>`/table `<td>`/toolbar wrapper → chrome (small); inside `<p>`/`<CardDescription>`/tooltip body/standalone explainer → prose (14px). Never put a full-sentence explanation at `text-[11px]`. See CLAUDE.md "Typography tiers" section.
- **Fix scripts run by the agent, not the operator.** When resolving a system issue requires a script run (DB migration, data backfill, cache invalidation, redeploy, schema repair), the agent executes it directly using the same env the app uses (`.env` + ADC). Do NOT hand the operator a "please run X" command. Announce the action briefly, run it, verify the outcome (re-curl the endpoint / re-query the table / confirm post-checks), and report. If the script fails (network, permissions, missing credential), surface the error verbatim and ask for the specific unblocker — don't silently fall back to delegating. Extend existing runners in `scripts/` in place; for Cloud SQL DDL use `scripts/apply-pending-migrations.ts` (connects via `@google-cloud/cloud-sql-connector` + IAM ADC, no proxy). Destructive-action restrictions still apply (no `DROP TABLE` / `git push --force` / etc. without explicit say-so). See CLAUDE.md "Fix scripts" section.
- **IPRoyal residential proxy is fallback-only, never pre-emptive.** SofaScore (`lib/settle/sources/sofascore.ts`) tries the direct Cloud Run request first and falls back to the IPRoyal pool ONLY on 403. Never route other sources (ESPN, football-data, Pinnacle, Gemini) through the proxy — they're not Cloudflare-blocked and it wastes metered IPRoyal bandwidth. Never use the proxy pre-emptively "just in case." Pool lives at `sessions/iproyal/proxies.txt` (gitignored, 168h TTL per sticky session). Cooldowns (30-min per-proxy on 403, 10-min global on direct 403) are intentional — don't shorten them. If you want the proxy for a non-SofaScore path, the right answer is almost always to fix that source properly, not to widen the proxy's blast radius. See CLAUDE.md "IPRoyal residential proxy — fallback only, never pre-emptive" section.
- **Explanatory copy is plain language with one concrete example.** Two paragraphs max for every tooltip / empty state / help text / glossary entry: (1) **headline** — one plain-English sentence in the bold first line, no acronyms, no field-of-study jargon ("Tests each strategy on bets it has never seen" beats "CPCV — Combinatorial Purged Cross-Validation"); (2) **body** — one short paragraph that combines the explanation with a concrete betting example (real providers like Pinnacle / NineWickets-Exchange / NineWickets-SB, real markets like 1X2 / Asian Handicap / BTTS, real-looking numbers like 100k BDT bankroll / +3.2% EV / 1,200 settled bets); (3) optional italic one-liner for picker-type entries answering "why pick this one?". Drop these jargon terms from body copy — use the right column instead: OOS → "on bets it has never seen", Sharpe → "smoothness of returns", p-value → "how likely this is just chance", Pareto frontier → "the trade-off line", confidence interval → "the believable range", math formulas → drop entirely. Acronyms (DSR, PBO, WRC, CPCV) are fine as column headers and pickers; the tooltip body must explain them without using the acronym again. NO "Why this matters" / "For your bets:" / "What you'll achieve:" labels. The glossary at `lib/lab/glossary.ts` is the reference implementation (`short` / `example` / `objective` fields; the legacy `long` field is deprecated and no longer rendered). See CLAUDE.md "Explanatory copy" section for the full rule + vocabulary cheatsheet.
